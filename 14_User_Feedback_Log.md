## FB-001 — 프라이머리 컬러 입력이 컬러피커가 아님

- 날짜: 2026-05-14
- 테스터: 유혜원
- 시나리오: 프로젝트 생성 직후 프라이머리 컬러 선택
- 원문: "프로젝트 만들고 프라이머리 컬러 선택하려하는데, 컬러피커가 아니라서 불편하다"
- 해석: 현재 입력 컨트롤이 텍스트 입력 또는 코드 직접 기입 방식으로 보이며, 시각적으로 색을 고르는 GUI 어포던스가 부족함.
- 관련 스펙: PRD §3.2 C1 ("Seed 입력 ... GUI로 직접 입력") — *컬러피커* 표현이 있으나 구체 컨트롤 규격은 미정. UI Spec(10)에서 컨트롤 정의 필요.
- 후속 결정 필요 사항:
  - 컬러피커 컨트롤 종류 (HSL 휠 / OKLCH 슬라이더 / 사각형 SV+휘도 슬라이더 / 사전 정의 팔레트 + Custom)
  - 헥스·OKLCH 텍스트 입력과의 병행 여부
  - 최근 사용 컬러 히스토리 노출 여부
- 우선순위: P1 (사용성 저하이지만 기능은 작동)
- 상태: open

## FB-002 — 엔터 시 컬러 값이 다르게 바뀜

- 날짜: 2026-05-14
- 테스터: 유혜원
- 시나리오: 프라이머리 컬러를 입력한 뒤 엔터 키 입력
- 원문: "컬러가 저장이 안되는건지.. 엔터누르면 다르게 바뀐다"
- 해석: 엔터 시 입력 값이 의도와 다르게 정규화되거나(예: 헥스가 가장 가까운 사전 정의 색으로 스냅), 입력 커밋 동작이 불명확함. 사용자는 자신의 값이 *손실*되었다고 느낌.
- 관련 스펙: UI Spec(10) — 입력 커밋 규칙(엔터/블러/디바운스) 미정의. PRD §3.2 C1의 *Map 자동 파생* 결과가 Seed 자체를 덮어쓰는 식으로 동작했을 가능성 있음.
- 후속 결정 필요 사항:
  - 엔터·블러·X초 디바운스 중 무엇이 "확정" 시점인지 명시
  - 입력값 정규화 시 사용자에게 "당신의 값 → 정규화된 값" 비교 노출 필요 여부
  - Undo 한 번에 입력 직전 상태로 복구 가능해야 함 (PRD §C6와 정합)
- 우선순위: P0 (작업 손실 또는 손실 인지 → 신뢰 저하)
- 상태: open

## FB-003 — 포커스링 자동 보정 버튼 부재

- 날짜: 2026-05-14
- 테스터: 유혜원
- 시나리오: 접근성 패널에서 포커스링 관련 미달 항목 확인
- 원문: "포커스링은 어떻게해야지 통과 되는지 모르겟어 ㅜㅜ 자동으로 맞춰지게 하는 버튼을 넣을 수는 없어?"
- 해석: 미달 사유 설명만으로는 어떤 토큰을 어떻게 조정해야 통과되는지 알 수 없음. PRD §3.2 F3 자동 보정 제안이 *대비비* 중심으로 기술되어 있어 포커스링까지 커버되지 않은 것으로 보임.
- 관련 스펙: PRD §3.2 F3 (자동 보정값 제안) — 포커스링 항목까지 확장 필요. Accessibility Spec(11)에 포커스링 통과 조건(두께·색·오프셋·콘트라스트)을 정의하고, UI Spec(10) 접근성 패널에 *"한 번에 통과시키기"* 버튼 자리 정의 필요.
- 후속 결정 필요 사항:
  - 자동 보정의 범위: 포커스링뿐 아니라 모든 WCAG/KWCAG 항목에 일괄 적용할지, 항목별로 제공할지
  - "한 번에 통과" 버튼이 변경하는 토큰 목록을 사전 미리보기로 보여줄지(권장)
  - 자동 보정이 디자이너의 의도(브랜드 컬러)를 침해하지 않도록 잠금 토큰은 변경 대상에서 제외하는 규칙
- 우선순위: P1 (수동 해결 가능하지만 학습 곡선·이탈 유발)
- 상태: open

  - 빈 상태(최근 작업 없음)의 UI
- 우선순위: P2 (사용성 개선, 기능 결함 아님)
- 상태: open

- 후속 결정 필요 사항:
  - 클릭 = 열기, 점 메뉴 = 복제/아카이브 등 부가 액션이라는 패턴 채택 여부
  - 키보드 접근 시 Enter도 동일 동작 보장 (a11y O-01)
- 우선순위: P1 (핵심 동선 차단)
- 상태: open

## FB-009 — 미달 항목별 자동/수동 가용성 명시 + 수동 조절 경로

- 날짜: 2026-05-15
- 테스터: 유혜원
- 시나리오: 접근성 패널에서 미달 항목 확인
- 원문: "수정할 수없는게 미달로 나오면 어떻게해? 사용자 입장에서 뭘해야하는지.. 알수가없는데 각각 미달항목을 자동/수동으로 조절할 수 있어야하지 않아?"
- 캡처된 사례:
  - **P-01 본문 텍스트 색상 대비비** — "위반 22건" (axe-core 결과). 어떤 노드들이 실패했는지 불명, 어떤 토큰을 바꿔야 하는지 모름, [보정 적용] 버튼 없음 (axe 결과는 fix 미생성).
  - **P-02 대형 텍스트 대비비** — 동일.
  - **O-03 포커스 가시 (외곽선)** — 2px · 2.66:1, "포커스 링 미달: 대비 2.66:1 (최소 3:1)". 두께는 통과지만 색 대비 미달이고, 현재 engine은 *ring 색 자동 보정을 의도적으로 스킵* (Primary 색 전체에 파급되므로). 사용자는 수동으로 조정할 UI도 없음.
- 해석: 두 종류의 미달이 섞여 있음 —
  1. 자체 룰이 만든 fail + autoFix 보유 (✓ 보정 버튼 노출 작동)
  2. axe-core가 만든 fail (fix 없음, 노드 위치도 안 보임)
  3. 자체 룰이 만든 fail이지만 fix 의도적 미생성 (focus ring 색 등) — UI에서는 그냥 "통과 불가"처럼 보임
- 후속 결정 필요 사항:
  - 각 항목에 **fixability 배지** — `자동 가능` / `자동 불가, 수동 필요` / `자동 보류 (다른 영역 영향)` 3종으로 명시
  - 자동 불가 항목에는 **"어떤 토큰을 조정해야 하는지" 가이드** + 해당 토큰으로 점프하는 링크(예: 우상 Customize 패널의 해당 슬라이더에 포커스)
  - axe 결과의 경우 **위반 노드 하이라이트** — 클릭하면 프리뷰에서 해당 요소에 outline 표시
  - "자동 보류" 항목에 *디자이너 의도 침해 경고 후 강제 보정* 옵션 — 옵트인 다이얼로그
  - O-03 ring 색 보정의 경우: ring color를 primary 외 alias 단계(예: focus.ring.color = primary.700)로 재매핑하는 보정 추가 검토
- 우선순위: P0 (도구 핵심 가치인 "접근성을 도구가 책임진다"가 사용자에게 전달되지 않음)
- 상태: open

## FB-010 — 비활성 프리셋의 어포던스 부족

- 날짜: 2026-05-15
- 테스터: 유혜원
- 시나리오: 좌측 스타일 프리셋 리스트에서 다크·미니멀·트렌디 등을 클릭 시도
- 원문: "프리셋은 아직 클릭이 안되는거같다"
- 해석: 의도된 동작 — PRD §3.1 D1에서 v0.1은 Default만, v0.2부터 9개 모두 활성화. 각 항목 하단에 "v0.2+" 라벨이 있지만 사용자가 "비활성" 상태임을 직관적으로 인지하지 못함.
- 후속 결정 필요 사항:
  - 비활성 항목 시각 약화 (opacity 0.5 + cursor not-allowed + 클릭 시 토스트 "v0.2에서 사용 가능")
  - 또는 v0.1 동안 비활성 프리셋을 *별도 섹션*("v0.2 예정")으로 분리해 클릭 가능한 영역과 구분
  - 또는 v0.1에서는 Default 외 프리셋을 *완전 숨김* 처리 (가장 단순)
- 우선순위: P2 (기능 결함 아님, 어포던스 개선)
- 상태: open

## FB-008 — 색상 대비비 자동 보정

- 날짜: 2026-05-14
- 테스터: 유혜원
- 시나리오: 접근성 패널에서 본문/헤딩/Primary 액션 대비비 미달 항목 확인
- 원문: "색생 대비비도 자동 보정되면 좋겟다"
- 해석: FB-003(포커스 링 자동 보정)에 이어 대비비 미달 항목도 클릭 한 번에 통과시키고 싶다는 요청.
- 관련 스펙: engine.ts의 `checkContrasts` + `findDarkerForContrast` 보정 알고리즘 (Sprint 1C T-016에서 구현), PRD §3.2 F3.
- 구현 상태:
  - P-01(본문×배경) 미달 시 → `seed.color.textBase`를 OKLCH L축에서 점진적으로 어둡게 조정해 4.5:1 통과 색 제안
  - P-02(헤딩) → 동일 textBase 보정
  - P-03(Primary 액션) → `seed.color.primary`를 어둡게 조정
  - "한 번에 보정" 버튼 1회 클릭으로 모든 대비비 미달 + 폰트 크기 + 한국어 폰트 누락 일괄 적용
- 후속 결정 필요 사항:
  - 사용자가 의도적으로 밝은 브랜드 컬러를 지정한 경우 보정이 디자인 의도 침해로 느껴질 수 있음 — 옵트인 확인 다이얼로그 필요 여부 (Accessibility Spec §5.5 옵트인 규칙)
  - 보정 후 *원본 색을 보존*하는 히스토리 UI (undo로 복귀 가능하지만, "보정 전 색 보기" 토글이 더 직관적일 수 있음)
- 우선순위: P1
- 상태: 방금 빌드 — 검증 대기 (refresh 후 동작 확인 필요)

## FB-007 — 모든 프로젝트 카드 클릭 = 열기

- 날짜: 2026-05-14
- 테스터: 유혜원
- 시나리오: 대시보드 "모든 프로젝트" 섹션의 카드 클릭
- 원문: "모든 프로젝트에서 그냥 카드 누르면 열기되면 좋겟음"
- 해석: 현재는 카드 표면 클릭으로 열리지 않고 별도 버튼이 필요한 듯. 카드 자체가 클릭 가능 영역이어야 함.
- 관련 스펙: UI Spec(10) 대시보드 인터랙션. FB-005와 동일한 패턴(최근 작업 카드 클릭 = 열기)이 모든 섹션에 일관되게 적용되어야 함.
- 후속 결정 필요 사항:
  - 카드 안 부가 액션 버튼(점 메뉴·아카이브 등)이 이벤트 버블을 막아 카드 클릭과 충돌하지 않도록 처리
  - 키보드 Enter·Space로도 열기 동작 (a11y O-01·O-13)
  - 호버 시 어포던스(보더 강조 또는 그림자)로 클릭 가능 명시
- 우선순위: P1 (FB-005와 동일 동선)
- 상태: open

## FB-006 — 아카이브 동작이 불명확

- 날짜: 2026-05-14
- 테스터: 유혜원
- 시나리오: 프로젝트 카드의 아카이브 버튼 클릭
- 원문: "프로젝트 아카이브 버튼 누르면 뭐가 되는지 알 수 없다.. 아카이브로 이동도 못하고.. 걍 프로젝트 사라진 사람 됨;"
- 해석: 현재 archiveProject는 projects 배열에서 단순 제거. "아카이브" 탭이나 보관함 화면이 없어서 사용자 입장에서는 *삭제*와 구분 불가. 게다가 사전 확인 다이얼로그도 없음 → 실수로 사라지면 복구 불가능.
- 관련 스펙: PRD §3.1 B3 ("프로젝트 삭제·아카이브" v0.2 예정), UI Spec(10) 대시보드.
- 후속 결정 필요 사항:
  - 아카이브 ≠ 삭제 분리 (Project에 `archivedAt` 필드 추가, 목록 필터)
  - 대시보드에 "아카이브 보기" 토글 또는 별도 탭
  - 아카이브 액션 전 확인 다이얼로그 + Undo (5초 토스트) 패턴 검토
  - 또는 v0.1 단계에서는 아카이브 UI 자체를 숨기고 v0.2에서 도입할지 결정
- 우선순위: P0 (작업 손실로 인식되는 동작)
- 상태: open
